## [Restore nested-codec name flattening (fix #156)](https://github.com/DitchOoM/buffer/pull/158)

**Maven Central:** [com.ditchoom:buffer:5.1.0](https://central.sonatype.com/artifact/com.ditchoom/buffer/5.1.0)

This release fixes a silent regression introduced by the v5.0.0 strip-and-rebuild of `buffer-codec-processor`: nested `@ProtocolMessage` data classes that shared a simple name across sibling sealed parents collided at KSP file-write time with `kotlin.io.FileAlreadyExistsException` (issue #156). Two same-named nested data classes in the same package — e.g. `CommandPayload.LedStateSet` and `ResponsePayload.LedStateSet` — both resolved to `LedStateSetCodec.kt` and the second write threw.

### Fixed

- **Nested-codec naming flattening restored.** The processor again prepends every enclosing type to the generated codec object name: `MqttPacket.SubAck` → `MqttPacketSubAckCodec`, matching the v4 convention from commit `923aaa48` that was lost in the v5.0 rebuild. Two same-named nested classes in sibling sealed parents now produce distinct codec files (`Issue156CommandLedStateSetCodec` and `Issue156ResponseLedStateSetCodec`) and round-trip independently.

### Migration

**Wire format is unchanged.** This is a code-shape change only.

If you upgraded from v4.x to v5.0.x and updated codec references from the flattened form (`MqttPacketSubAckCodec`) to the simple form (`SubAckCodec`) to satisfy the v5.0 generator, you need to flatten those references again when moving to v5.1.0. The convention is: every enclosing type prepends to the codec object name.

```kotlin
// v4.x: works
MqttPacketSubAckCodec.decode(buffer, ctx)

// v5.0.x: broken
SubAckCodec.decode(buffer, ctx)

// v5.1.0: works (matches v4.x)
MqttPacketSubAckCodec.decode(buffer, ctx)
```

If your project never updated for v5.0's simple-name form (e.g. you stayed on v4.x), no code change is needed.

### Regression coverage

The PR adds `Issue156NestedNameTest`, which encodes the exact bug shape from the issue (two same-named nested data classes in sibling sealed parents) and asserts both distinct codec object names and independent round-trip. Eleven existing MQTT test files were renamed for the restored convention; no assertion logic changed.
